# 三种估值：把 woody “实时估值”讲清楚（含校准值源码推导）

这篇文章只做一件事：把 woody（palmmicro/woody）里 **Official / Fair / Realtime** 三种估值的“真实实现”讲清楚，尤其是第三种 **RealtimeNetValue（实时估值）** 里最容易被误读的“校准值（calibration / factor）”。

> 本文基于源码阅读得出结论（关键文件）：`php/stock/qdiiref.php`、`php/stock/holdingsref.php`、`php/stock/fundpairref.php`、`php/sql/sqldailytime.php`、`php/stock/stockref.php`、`php/ui/calibrationhistoryparagraph.php`。

---

## 0. 先统一概念：woody 的“三种估值”不是三套完全独立的公式

在 woody 里，“三种估值”对应的是三个 getter（名称一致，但实现会因基金类型不同而不同）：

- `GetOfficialNetValue()`：**官方估值**（更准确地说，是“用官方净值日期/官方收盘数据为锚点”算出来的估值），即我程序里面的**静态ETF估值**
- `GetFairNetValue()`：**参考估值**（用更靠近当前时刻的价格/汇率重算一次，并加上一个“官方校正项”保证与官方锚点连续）
- `GetRealtimeNetValue()`：**实时估值**（当底层参考标的存在“实时替身”（配对交易模型）时，用替身推算价格，再做一次 Fair 的那套逻辑）即我程序里面的**实时ETF估值**

这三者在源码里主要有两条路径：

1) **QDII 单参考标的型**（“基金 ↔ 指数/ETF”的直接映射）：`php/stock/qdiiref.php`  
2) **QDII 多持仓/混合型（Holdings）**（“基金持仓由多个 ETF/股票构成”）：`php/stock/holdingsref.php`

黄金 LOF（例如 `SZ160719`）在 woody 里走的是 **HoldingsReference 路线**，而不是简单的单指数 QDII 公式。

## 1. 最核心的“校准值”到底是什么？（以 GLD ↔ hf_GC 为例）

页面：`…/res/calibrationhistorycn.php?symbol=GLD` 的“校准值”，对应数据库表 `calibrationhistory` 的 `close` 字段；页面只是把它格式化输出（`php/ui/calibrationhistoryparagraph.php`）。

### 1.1 校准值（factor）的计算公式

源码入口：`php/stock/fundpairref.php` 的 `FundPairReference::CalcFactor($pair, $net, $date)`：

- 先取配对标的价格（或净值）：`fPair = float(pair)`
- 再取本标的价格（或净值）：`fNet = float(net)`
- 如有跨币种（存在 `$this->cny_ref`），会把 `fNet` 换算到与 `fPair` 同币种后再比（方向由是否 A 股决定）
- 返回：

```
factor = fPair / fNet
```

 `GLD ↔ hf_GC`（USD 对 USD，无汇率换算）：

```
factor(GLD) = Price(hf_GC) / Price(GLD)
```

### 1.2 “实时校准”怎么触发？是每分钟一次吗？

不是后台定时任务。触发机制是：**当网页请求导致两边行情被刷新**，且两边行情满足严格对齐条件时，就把这一次的 `factor` 记为当天的一个样本。

对齐条件来自 `php/stock/stockref.php` 的 `CheckAdjustFactorTime($etf_ref)`：

- 两边都有数据、且都不在盘前/盘后（`IsExtendedMarket()`）
- 日期相同
- **HH:MM 完全相同**（注意：是分钟级严格相等，不是“差不多”）

并且 `FundPairReference` 在构造阶段就会做判断，满足就调用 `AverageCalibraion()` 写样本（`php/stock/fundpairref.php`）。

之所以你看到它“像每分钟一次”，是因为抓新浪行情有 **一分钟级限频/本地缓存**（`StockNeedFile(..., SECONDS_IN_MIN)`）：同一分钟内重复请求通常拿不到“新一分钟时间戳”的行情，自然也不会产生新的样本。

另外，woody 里“非盘前盘后”的判定窗口对美股/美期写死为 **09:30–16:00（美东）**（`IsExtendedMarket()` 使用这个窗口来判断是否 extended）。

补充一个你关心的点：**“用户访问触发校准”并不是额外条件**，它只是“让行情去刷新”的触发器；真正能不能写入样本，仍然要满足“同日同分钟 + 常规时段”的严格对齐条件。也正因为是“访问驱动”，当 A 股长假（比如春节）大家不看盘、网站没人访问时，**校准采样次数会明显下降**——这恰恰是合理的：A 股不交易时再去算“期货校准实时估值”本身就没有实际意义。

### 1.3 它不存分钟明细，那 16:00 怎么算均值？

它不是等到收盘再算，而是每次触发时做“运行中均值更新”。

`php/sql/sqldailytime.php` 的 `CalibrationSql::WriteDailyAverage()`：

- 当天第一条：直接插入 `avg = factor`，`num = 1`
- 之后每条：增量更新算术均值：

```
newAvg = (num * oldAvg + factor) / (num + 1)
num += 1
```

还有一个细节：如果 `abs(oldAvg - factor) <= MIN_FLOAT_VAL`，它会认为“没有变化”，不会让 `num+1`（避免超高频重复样本把 `num` 撑爆）。

因此：数据库里 `calibrationhistory.close` **永远都是“截至当前时刻的当日均值”**，不需要保留每分钟明细，更不需要到 16:00 再回放计算。

## 2. 三种估值在 woody 里的真实含义（按源码口径）

下面用“Holdings 型”和“QDII 单标的型”分别讲清楚。

---

## 2.1 Holdings 型（多持仓）——以 SZ160719 这类为代表

源码：`php/stock/holdingsref.php`。

### (1) OfficialNetValue：官方估值（锚定官方净值日）

`HoldingsReference::GetOfficialNetValue()` 的核心流程：

1. 选择一个“官方锚定日期” `officialDate`（逻辑在 `GetOfficialDate()`，会考虑 A 股基金/外汇收盘是否可用等）
2. 计算一个“模型估值” `modelOfficial = _estNetValue(officialDate, bRealtime=false)`
3. 读取数据库里该日真正的基金净值 `navOfficial = netvaluehistory[officialDate]`（如果存在）
4. 记录一个“官方校正项”：

```
officialAdjust = navOfficial - modelOfficial
```

并把 `modelOfficial` 写到 `fundest`（用于展示/回溯）。

> 注意：OfficialNetValue 在这里是“模型估值”（`modelOfficial`），不是直接等于基金公司发布的 `navOfficial`。`navOfficial` 被用来形成 `officialAdjust`，从而让后续 Fair/Realtime 估值在数值上与官方锚点“对齐且连续”。

### (2) FairNetValue：参考估值（用更新的价格/汇率重算，再加校正项）

`HoldingsReference::GetFairNetValue()`：当它判断“确实需要更实时”（比如外汇/持仓标的日期已经更新）时：

1. 先确保已算出 Official（拿到 `officialAdjust`）
2. 用当前时刻的数据再算一次：

```
modelFair = _estNetValue(date=false, bRealtime=false)   // 用当下价格/汇率
fair = modelFair + officialAdjust
```

因此 Fair 并不是“只做汇率替换”的简化公式，而是**完整重算持仓变化**，最后再用 `officialAdjust` 对齐官方锚点。

### (3) RealtimeNetValue：实时估值（把“可实时推算的持仓价格”替换掉）

`HoldingsReference::GetRealtimeNetValue()`：只有当 `UseRealtimeEst()` 为真（也就是持仓里存在 `FundPairReference` 这种可用配对模型推算实时价的标的）才会启用。

它与 Fair 的区别只有一处：重算时传 `bRealtime=true`，对持仓里的某些标的用 `EstFromPair()` 替代直接取 `GetPrice()`：

```
modelRealtime = _estNetValue(date=false, bRealtime=true)
realtime = modelRealtime + officialAdjust
```

一句话总结：Realtime 不是“另一套公式”，而是“同一套持仓估值公式 + 用配对交易模型把部分持仓价格替换成更实时的推算值”。

## 2.2 QDII 单参考标的型（直接映射）

源码：`php/stock/qdiiref.php`。

它的核心是一个因子 `fFactor`：

- 当“官方净值”当天准备好时，会用 `AdjustFactor()` 重新校准 `fFactor`，使模型与真实净值对齐：

```
fFactor = est * CNY_close / fund_nav
```

（源码函数：`QdiiGetCalibration($est, $cny, $netValue)`）

- 之后任何估值都走：

```
val = (est * CNY) / fFactor
val = AdjustPosition(val)    // 用仓位与基准净值做线性混合
```

其中 `est` 的取法分三种：

- Official：优先用数据库里该日的 `est_ref` 净值/收盘（或 holdings 估值）
- Fair：当 `est_ref` 日期或汇率日期已变动，用更新后的 `est`
- Realtime：如果 `est_ref` 自己也有“配对模型”（`GetRealtimeRef()` 存在），则用 `est_ref->EstFromPair()` 得到更实时的 `est`

## 3. 这套系统的“最容易误读点”清单

1) **“实时估值”不是每分钟定时任务**：它是“请求驱动 + 分钟对齐”触发写样本。  
2) **校准值不需要存分钟明细**：`calibrationhistory.close` 本身就是增量均值。  
3) **Fair 不是只替换汇率**：Holdings 型会重算持仓变化，再加 `officialAdjust`。  
4) **Realtime 不是另一套估值公式**：它只是“同一套估值 + 用配对模型替换部分输入价格”。  
5) **SZ160719 这类是 Holdings 型，不是单指数 QDII 简化公式**：用错模型会导致你以为“某些简化公式正确”，但其实只是“恰好在某些日子近似”。

## 5. 本系统的三种实时估值实现

以上讲的是 woody 系统的估值逻辑。我们自己的系统（LOFarb）有三种实时估值，它们的核心公式和静态估值完全一样，区别在于**价格数据源不同**。

### 5.1 三种实时估值概览

| 实时估值列 | 位置 | 数据来源 | 适用场景 |
|-----------|-----|---------|---------|
| ETF估值 | 最右边第一列 | 真实ETF实时价格 | 最准确，优先参考 |
| 期货校准 | 最右边第二列 | 期货实时价格 + 校准值 | 24小时有数据，美股休市时用 |
| 纯期货映射 | 最右边第三列 | 期货实时价格（直接用） | 解决ETF对冲流动性不足、滑点太大 |

---

### 5.2 ETF估值（最准确）

**数据来源**：真实的ETF实时价格（如GLD、USO、XOP等）

**计算公式**：
```
实时估值 = T-1日净值 × [1 + 仓位 × (ETF实时价/T-1日ETF价 × 汇率变动 - 1)]
```

**特点**：
- 和静态估值公式完全一样，只是把T日价格换成**实时价格**
- 通过JavaScript实时从新浪/IB获取数据更新
- 最准确，但15:00美股开盘前可能数据不完整

---

### 5.3 期货校准

**数据来源**：期货实时价格 、校准值

**校准值来源**：
- 每天从woody网页爬取
- 不是拟合的，每天都在变
- 黄金校准值≈10.9，原油校准值≈0.82（示例值）

**核心步骤**：
1. 先用**校准值**把期货价格映射到ETF价格空间：
   ```
   等价ETF价 = 期货实时价 / 校准值
   ```
2. 然后用和**ETF估值一样的公式**计算

**计算公式**：
```
等价ETF价 = 期货实时价 / 校准值
实时估值 = T-1日净值 × [1 + 仓位 × (等价ETF价/T-1日ETF价 × 汇率变动 - 1)]
```

**特点**：
- 24小时都有期货数据，不会断
- 需要校准值准确才能保证估值准确

---

### 5.4 纯期货映射（解决ETF流动性问题）

**数据来源**：期货实时价格（直接用，不经过校准值）

**计算公式**：
```
实时估值 = T-1日净值 × [1 + 仓位 × (期货实时价/T-1日期货结算价 × 汇率变动 - 1)]
```

**注意**：
- 这里用的是**T-1日期货结算价**作为基准价，不是收盘价

**设计目的**：
- 解决ETF对冲时**流动性不足、滑点太大**的问题
- 纯期货市场流动性更好，滑点更小

**特点**：
- 最简单，不需要校准值
- 直接用期货价格作为底层价格
- 专为期货对冲设计，避免ETF的流动性问题

---

### 5.5 三种实时估值的适用场景

| 场景 | 推荐使用 | 原因 |
|-----|---------|-----|
| 美股交易时段 | ETF估值 | 最准确 |
| 美股休市时段 | 期货校准 | 24小时有期货数据 |
| 做期货对冲交易 | 纯期货映射 | 期货市场流动性好，滑点小 |
| 校准值不准时 | 纯期货映射 | 不需要校准值 |

## 6. 终极揭秘：Woody 实时估值的数学魔法（常量折叠）

我们在对接 Woody API 时发现，他并不返回基准日的 ETF 价格和基准日汇率，而是返回了 `calibration` (校准值) 和 `hedge` (对冲值)。这背后隐藏着极高明的**“常量折叠 (Constant Folding)”**与降维打击思想。

### 6.1 原始长公式的痛点
我们常规的实时估值推导公式是（正如 5.2 节所示）：
```text
实时估值 = 基准净值 × [ 1 + 仓位 × (实时ETF × 实时汇率 / (基准ETF × 基准汇率) - 1) ]
```
在这个公式里，盘中运算必须时刻知道“基准ETF”和“基准汇率”。这让前端计算变得非常臃肿，需要时刻携带昨天的历史账本。

### 6.2 Woody 的降维：Calibration (校准值)
Woody 发现，在 T 日盘中运算时，**基准净值、基准ETF、基准汇率，这三个数全都是昨天就已经定死的常量！**
于是他定义了一个降维常量 **Calibration（校准值）**：
```text
Calibration = (基准ETF × 基准汇率) / 基准净值
```
*(物理意义：每 1 份 LOF 净值里，包含了多少人民币价值的底层 ETF)*

代入原始公式化简后，分母奇迹般地完全抵消了：
```text
实时估值 = 基准净值 × (1 - 仓位) + (仓位 / Calibration) × 实时ETF × 实时汇率
```

**正如 Woody 原话中所说：**
> “前半部分是不变的现金仓位，后半部分是变化的股票持仓。API接口的使用者只需要自行拿到XOP的价格（注：即实时ETF），就能计算对应的SZ162411净值EST。”

### 6.3 终极杀器：Hedge (对冲值)
在上述化简公式中，后半部分的系数是 `仓位 / Calibration`。
Woody 将这个倒数单独提炼出来，通过 API 传递给了我们，命名为 **Hedge（对冲值）**：
```text
Hedge = Calibration / 仓位
```

**Hedge 的绝对物理意义：**
它代表的是**纯粹的物理兑换比例**！即：**1 股美股 ETF，恰好等于 Hedge 股 A股 LOF！**

将 Hedge 代入，实时估值公式变成了极致简化的终极形态：
```text
实时估值 = 基准净值 × (1 - 仓位) + (实时ETF × 实时汇率) / Hedge
```

## 7. 终极揭秘其二：指数基金的三轨合一与数学等价性

在实际盘中，对于纯指数类基金（如 161130 纳指、161125 标普），我们会观察到一个神奇的现象：ETF实时估值、期货校准估值、纯期货映射估值，三者算出的溢价率往往高度一致甚至一模一样。

这并非偶然，而是底层因子提取极其精确时展现出的**“数学等价性”**。

### 7.1 为什么“魔法公式” = “原始涨跌幅矩阵”？

如果我们把上一节推导出的 Hedge 物理定义代回魔法公式进行逆向展开：

```
Hedge = (基准日ETF单价 × 基准日汇率) / (基准日LOF净值 × 仓位)
```

将其代入魔法公式：

```
实时估值 = 基准净值 × (1 - 仓位) + (仓位 / Hedge) × (实时ETF × 实时汇率)
```

在纸上进行代数展开和约分（把基准日净值和仓位约掉），你会发现，它化简到最后恰恰就是最原始的涨跌幅矩阵公式：

```
实时估值 = 基准净值 × [ 1 + 仓位 × (实时ETF/基准ETF × 实时汇率/基准汇率 - 1) ]
```

**结论**：只要 Woody 提供的 Hedge 取值精确，不论是用简化版的“魔法公式”算，还是退回“T-1矩阵涨跌幅”算，数学上它们是同一个恒等式，结果必然完全一致。

### 7.2 为什么“期货校准” = “纯期货映射”？

纯期货的对比逻辑是：
```
实时估值变化 ∝ 实时期货 / 基准日期货
```

而期货校准的逻辑是：先把期货剔除升水变为现货，再去和基准日现货对比：
```
实时估值变化 ∝ (实时期货 / 校准升水率) / 基准日现货指数
```

由于基准日现货指数本身往往就等于 `基准日期货 / 校准升水率`。如果你把公式里分子分母同时除掉校准升水率（比如 1.003），你会发现 1.003 被完美约分抵消了！整个公式瞬间坍缩回了 `实时期货 / 基准日期货`。

**结论**：只有当基金面临换月展期，或者升贴水率突然发生巨变（比如从 1.003 暴增到 1.006）时，现货校准与纯期货的误差才会显现。在大部分平稳期，它们在数学上必然收敛于同一个值。

### 7.3 系统架构的最终抉择：大道至简

基于上述数学等价性，我们在处理指数类基金的“期货校准估值”时，坚决摒弃了“把实时期货转为现货，再用现货推导美股ETF，最后再套用魔法公式”的二次校准（脱裤子放屁）做法。

在本系统的代码实现中（如 LOF035 及 JS 运算逻辑），对于指数基金的期货校准：

1. **UI 层面**：贴心地计算并展示相当于多少**校准ETF测试价**，供用户直观参考。
2. **底层计算层面**：直接暴力采用现货指数（如 NDX）对比基准日现货指数的原始涨跌幅模型。代码更少，逻辑更正宗，从根本上杜绝了浮点数多次乘除带来的精度丢失与逻辑撕裂。
