# 估值算法详解

## 估值方法概述

LOF 基金套利监控系统使用多种估值方法计算 LOF 基金的实时估值，为套利策略提供决策依据。系统实现了三种主要的估值方法：静态官方估值、期货校准估值和期货原生估值。

## 估值方法详解

### 1. 静态官方估值 (Static Official Valuation)

**原理**：基于基金的 T-1 日净值，结合当前 ETF 和汇率的变化，计算当前的估值。在最新的 `LOF012_calculate_static_valuation.py` 引擎中，全面采用**“双轨推演”**架构。

**计算公式（双轨引擎）**：
- **极速魔法通道（优先）**：利用 Woody API 返回的常量折叠因子 `Hedge`，进行 O(1) 极速推演。
  `静态估值 = T-1日净值 × (1 - 仓位) + (T日主锚点ETF价格 × T日汇率) / T-1日Hedge`

- **矩阵降级通道（兜底）**：当 Woody API 宕机或因子缺失时，自动降级退回传统的“一篮子 ETF + 权重”计算矩阵。
  `静态估值 = T-1 日净值 × [1 + 仓位 × (Σ(T日ETF / T-1日ETF × 权重) × (T日汇率 / T-1日汇率) - 1)]`

**适用场景**：
- 最基础的核心估值算法，用来核对估值算法的准确性
- 是实时估值的基础


### 2. 纯 ETF 实时估值 (ETF Realtime Valuation)

**原理**：在美股交易时段，直接使用真实的底层 ETF 实时价格进行推演。

**计算公式（大一统魔法捷径）**：
`实时估值 = T-1日净值 × (1 - 仓位) + (ETF实时价 × 实时汇率) / API_Hedge`

**适用场景**：
- 美股正常交易时段，数据最准确，优先参考。


### 3. 期货校准实时估值 (Futures Calibration Valuation)

**原理**：通过 Woody API 提供的校准值 (`calibration`)，将 24 小时交易的期货价格等效映射为 ETF 价格，再代入估值。

**计算公式（大一统魔法捷径）**：
`等价ETF价 = 期货实时价 / API_Calibration`
`实时估值 = T-1日净值 × (1 - 仓位) + (等价ETF价 × 实时汇率) / API_Hedge`

**适用场景**：
- 亚洲盘或美股盘前时段（ETF 无实时行情时），利用大宗期货不间断的特性进行估值。


### 4. 纯期货映射估值 (Futures Native Valuation)

**原理**：解决部分挂钩 ETF（如 XOP）流动性不佳或滑点过大的问题，直接利用 T-1 日的期货结算价物理反推出专属的 `Futures_Hedge`，无缝接入魔法引擎。

**计算公式（逆向物理推导魔法）**：
`物理反推 Futures_Hedge = (T-1日期货结算价 × T-1日汇率) / (T-1日净值 × 仓位)`
`实时估值 = T-1日净值 × (1 - 仓位) + (期货实时价 × 实时汇率) / Futures_Hedge`

**适用场景**：
- 专为实盘后台轮动套利设计，充当极速、低滑点的交易对冲锚点。

**💡 系统架构落地**：
以上魔法公式已被完整剥离并封装在 `arbcore/calculators/valuation_math.py` 中，静态估值（012）与动态盘中估值（02 后端推演及 03 前端沙盘）现已全面调用此核心数学引擎，实现了代码的绝对同源与极致性能！


## 估值数据来源

### 1. 基金净值历史数据

- **来源**：基金公司官方
- **更新频率**：每日更新（通常在交易日结束后傍晚）
- **存储位置**：`lof_data` 表的 `nav` 字段

### 2. 基金价格历史数据

- **来源**：新浪财经
- **更新频率**：每日更新（在交易日结束后）
- **存储位置**：`lof_data` 表的 `price` 字段

### 3. 美股ETF 数据

- **来源**：woody API 、新浪财经
- **更新频率**：每日更新 
- **存储位置**：`etf_daily_prices` 表（纯净指数存入 `index_daily` 表）

### 4. 汇率数据

- **来源**：国家外汇管理局、新浪财经
- **更新频率**：每日更新
- **存储位置**：`exchange_rate` 表

### 5. 期货数据

- **来源**：新浪财经/CME API
- **更新频率**：每日更新 
- **存储位置**：`futures_daily` 表

## 估值计算流程

### 1. 数据准备

1. **获取基准日和估值日的基础数据**：
   - 基金的仓位
   - 基金净值
   - ETF 价格
   - 中间价汇率
   - 期货价格


2. **获取当前数据**：
   - 基金的仓位
   - 当前夜盘的 ETF 价格
   - 当前汇率（中间价、或者在岸价）
   - 当前期货价格


### 2. 静态官方估值，计算估值日/基准日

1. 计算ETF 价格变化率
2. 计算汇率变化率
3. 根据仓位、变化率调整 T-1 日净值
4. 得到静态官方估值

### 3. 实时估值，计算当日/基准日

1. 计算ETF 价格变化率
2. 计算汇率变化率
3. 根据仓位、变化率调整 T-1 日净值
4. 得到实时估值

### 5. 估值结果处理

1. 存储估值结果到数据库
2. 计算溢价率
3. 检测套利机会
4. 更新监控界面

## 校准值和对冲值计算

### 1. 校准值 (Calibration Value)

**原理**：LOF净值*校准值= 美股ETF净值*汇率

**存储位置**：`fund_daily_factors` 表的 `calibration` 字段

### 2. 对冲值 (Hedging Value)

**原理**：用于对冲，= 校准值/仓位**

**存储位置**：`fund_daily_factors` 表的 `hedge` 字段

### 3. 期货校准值 (Futures Calibration Value)

**原理**：通过期货价格与 ETF 价格的关系计算得出，用于调整基金估值。

**计算步骤**：woody网页在美股交易时段，按照某个规则计算期货价格和 ETF 价格数据的比率，确定校准因子


## ⚙️ 对冲值算法
### 1. 对冲数量 (Hedge Quantity) 的双轨推演算法
在量化沙盘中，当系统预设固定的人民币本金时，必须**极其精准地推算出敞口（Delta）闭合状态下，中美两市所需的委托股数。系统针对不同类型的资产设计了两套独立的逆向物理推导模型，其核心锚点全部建立在 API 解析而来的物理 `对冲值 (Hedge)` 之上。**

#### 1.1 纯ETF与指数类（如 162411, 161127, 161125, 161130）
**算法逻辑：美股定 A 股**。由于此类资产成分高度单一，等效追踪效应极强，须保证美股股数的整数下探精准。
1. **计算临时拟合股数**：`临时数量 = 投入金额 / LOF盘中实时交易价`
2. **锁定美股实际委托头寸**：`美股对冲数量 = ROUND(临时数量 / 对冲值)` （四舍五入，保底 1 股）
3. **反推锁定A股委托头寸**：`LOF实际交易数量 = ROUND((美股对冲数量 × 对冲值) / 100) × 100` （A股受限100股倍数规则，依此凑整）

#### 1.2 商品类（如 160719 黄金, 160723 原油）
**算法逻辑：A股定美股**。大宗商品 LOF 底层常由全球一篮子资产拼接转换，A 股为实际执行主端，需先强制凑齐境内手数，后用海外衍生品兜底。
1. **锁定A股实际委托头寸**：`LOF实际交易数量 = ROUND((投入金额 / LOF盘中实时交易价) / 100) × 100`
2. **被动派生美股委托头寸**：`美股对冲数量 = ROUND(LOF实际交易数量 / 对冲值)` （四舍五入，保底 1 股）
*(注：大宗商品本身缺乏直接对冲比参数，其底层物理“对冲值”将在 012 构建中，通过 `Woody动态校准值 ÷ 基金实际合约仓位比例` 重建并自动固化。)*

根据目前代码中的实现逻辑，A股 LOF 价格和 IB美股 ETF 价格采用了不同的取价策略，具体如下：

1. A股 LOF 价格：优先采用【卖一价 (Ask 1)】，兜底采用【最新成交价】
无论是通过哪个行情引擎（银河QMT、通达信、国金QMT，还是新浪API兜底），系统在获取 LOF 价格时，都严格遵循了 优先取卖一价 的防守型原则。

常规情况：直接使用盘口的 卖一价 (Ask 1)。因为对于溢价套利而言，按卖一价买入 LOF 才是你能立刻拿到筹码的真实成本，这样计算出的溢价率更加保守和真实。
涨停或无卖盘：代码中有一层防呆机制，如果遇到 卖一价 == 0（比如基金涨停封板，卖盘被清空），此时系统会自动降级，兜底使用 最新成交价 (Last Price / Now) 作为计算基准。
(代码出处：LOF02_fetch_trade_data.py 中的 LOFPriceReader 类，各引擎回调中均有 price_to_use = float(snap.get('Sell1', 0))，if price_to_use == 0: price_to_use = float(snap.get('Now', 0)) 等类似判断)

2. IB 的美股 ETF 价格（夜盘）：采用【买一价 (Bid 1)】
在系统展示和推演时，从 IB API (外盘) 拿到的美股 ETF 价格采用的是 买一价 (Bid)。

在前端轮询 IB 数据更新的 JS 代码中，专门提取了 bid 字段映射到页面及沙盘的计算中： prices.GLD && prices.GLD.bid ? prices.GLD.bid.toFixed(2) : '-'
取“买一价”是因为在夜盘对冲（卖出/做空底层 ETF）时，你是以 Bid 价把筹码抛给买方的，这也是一种贴近实际交易滑点的防守型取价方式。
总结：

LOF 实时价格：卖一价（涨停时降级为最新成交价）。
IB ETF 实时价格：买一价。
这套取价逻辑在数学上构成了**“买入 A股 LOF (看卖一) + 卖空 美股 ETF (看买一)”** 的严谨对冲闭环，把盘口点差（Spread）的摩擦成本已经包含在溢价率的计算当中了，因此面板上显示的溢价率是没有水分的“净利润率”。
