# 后端数据引擎架构与流程 (新版：数据库驱动)

## 1. 架构总览：数据库驱动的两阶段流水线

新版后端彻底抛弃了繁杂的 CSV 文件管理，全面转向以 **SQLite 数据库为核心** 的数据处理架构。这套架构保证了数据的一致性、稳定性和可维护性，将数据采集与数据计算完全解耦。

- **`LOF011_daily_updater.py` (数据采集器)**：作为流水线的起点，它负责从所有外部数据源（Woody API、外汇管理局、新浪、东方财富等）抓取原始数据，并将其**原封不动地存入 SQLite 数据库**的各个标准表中。此模块是唯一与外界网络交互的模块。
- **`LOF012_calculate_static_valuation.py` (静态估值计算器)**：作为流水线的终点，它是一个**纯粹的计算引擎**，不执行任何网络请求。它从 SQLite 数据库中读取所有需要的数据，在内存中进行高效的关联和计算，最终将包含静态估值、溢价率等完整信息的对账表，写回数据库中供前端调用。

---

## 2. 核心数据枢纽：SQLite 数据库 (`arb_master.db`)

所有的数据，无论是原始的还是计算后的，都集中存储在 `arb_master.db` 数据库中。关键表结构如下：

- **`raw_api_data`**: 存储从 Woody API 获取的原始 JSON 报文，作为数据湖的源头，用于追溯和调试。
- **`exchange_rate`**: 存储每日人民币中间价。
- **`etf_daily_prices`**: 存储所有海外 ETF（如 GLD, USO, XOP, ^GLD-EU）和指数（.INX, .NDX）的每日价格。
- **`futures_daily`**: 存储所有期货（如 GC, CL, NQ）的每日结算价和校准值。
- **`lof_data`**: 存储所有 A 股 LOF 基金的每日官方净值（NAV）和收盘价。
- **`fund_daily_factors`**: 存储每个基金每日的私有因子，如仓位、对冲值等。
- **`fund_basket_weights`**: 存储基金的底层资产构成及权重。
- **`lof_history_{fund_code}`**: **最终成品表**。每个基金一张，存储了经过计算的、包含所有静态估值和对账信息的完整历史数据，直接供前端渲染。

## 3. `LOF011_daily_updater.py` 执行流程 (数据采集)

专职数据采集：

这个新脚本现在是唯一的数据入口。
它的任务是：从所有外部源（Woody API、外汇管理局、新浪、东方财富）抓取最新数据，然后分门别类地存入数据库的各个标准表中。
它内置了防封禁机制，当天成功获取过的数据不会重复抓取。

1.  **获取 Woody API 数据**:
    *   检查数据库，如果今天已经成功获取过，则直接跳过，避免IP被封。
    *   若未获取，则调用 API，将返回的原始 JSON 存入 `raw_api_data` 表。
    *   **智能解析** JSON，将打平后的基金因子、ETF价格、持仓权重等信息，分门别类存入 `fund_daily_factors`, `etf_daily_prices`, `fund_basket_weights` 等标准表中。
2.  **获取宏观市场数据**:
    *   从外汇管理局获取最新人民币中间价，存入 `exchange_rate` 表。
    *   从新浪获取主要期货（GC, CL, NQ, ES）的 T-1 日结算价，存入 `futures_daily` 表。
3.  **获取 LOF 行情数据**:
    *   遍历配置文件中的所有基金。
    *   从东方财富获取历史净值（NAV）。
    *   从新浪获取历史收盘价。
    *   将净值和收盘价按日期合并，存入 `lof_data` 表。

## 4. `LOF012_calculate_staticvaluation.py` 执行流程 (估值计算)
-纯粹的计算引擎：

这个脚本现在不再进行任何网络请求，变成了一个纯粹的“计算器”。
它从数据库中读取 01 号脚本准备好的所有原材料，在内存中进行高效的关联和数学计算。
计算完成后，将包含所有静态估值、溢价率的最终结果，写回数据库中供前端网页渲染。

1.  **遍历基金**: 依次处理 `lof_config.yaml` 中配置的每一只基金。
2.  **数据加载**: 对于每只基金，从数据库中通过 `JOIN` 操作，一次性加载其所需的所有数据：
    *   自身的净值和收盘价 (from `lof_data`)
    *   每日汇率 (from `exchange_rate`)
    *   每日仓位等因子 (from `fund_daily_factors`)
    *   所有底层 ETF 的每日价格 (from `etf_daily_prices`)
    *   相关期货的每日结算价 (from `futures_daily`)
3.  **核心矩阵运算**:
    *   在 Pandas DataFrame 中，通过历史回溯窗口，找到每个交易日 T 所对应的有效基准日 T-1。
    *   基于 T-1 的数据（净值、汇率、ETF价格、期货价），和 T 日的数据，进行矩阵运算，批量计算出 T 日的各项静态估值。
    *   支持**ETF静态估值**、**期货静态估值**、**纯指数静态估值**等多种模型的并行计算。
4.  **结果入库**:
    *   将计算完成的、包含所有估值和误差列的完整 DataFrame，直接覆盖写入到对应基金的 `lof_history_{fund_code}` 表中。
    *   这张表就是前端页面的最终数据源。

## 5. 异常应对指南

| 异常现象 | 排查思路 |
|---|---|
| **控制台卡在 `获取 Woody API 数据`** | Woody API 可能暂时不可用。脚本有超时和重试机制。如果长时间卡死，可检查网络或稍后重试。由于有数据库缓存，当天成功一次后不会再请求。 |
| **`LOF012` 脚本报错 `KeyError` 或 `column not found`** | 通常是因为 `LOF01_daily_updater.py` 未能成功抓取到某个必要的数据（如某个ETF的价格）。请检查 `01` 脚本的日志，确认所有数据是否都已成功入库。 |
| **某天没有计算出估值** | 该交易日或其 T-1 基准日，在数据库中缺失了计算所需的某个关键数据（如净值、汇率或某个底层ETF价格），导致计算跳过。 |
| **新增了一只基金，但页面没显示** | 1. 确保在 `lof_config.yaml` 中已正确添加该基金的配置。 2. 依次运行 `LOF01_daily_updater.py` 和 `LOF012_calculate_valuation.py`，确保其数据已采集并计算。 3. 检查 `01` 和 `012` 的日志，确认没有报错。 |
