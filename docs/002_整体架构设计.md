# 系统架构设计

## 系统概述

### 设计目标

1. **实时性**：毫秒级行情更新，捕捉瞬间套利机会
2. **准确性**：三重估值体系交叉验证，降低单一数据源风险
3. **稳定性**：多数据源自动切换，单点故障不影响整体
4. **可扩展性**：模块化设计，易于添加新基金或数据源


## 整体架构

LOF 基金套利监控系统采用三层架构设计，实现了核心功能与业务逻辑的分离，提高了系统的可维护性和可扩展性。

```
+---------------------+
| 应用层 (Applications) |
+---------------------+
|  |
|  +-------------------------+

|  +-------------------------+

|  +-------------------------+

|  +-------------------------+
```

## 核心层 (arbcore)

核心层提供了系统的基础功能，为应用层提供服务。

### 数据获取模块 (arbcore/fetchers)

- **data_fetcher.py**：统一的数据获取接口，负责从多个数据源获取数据
  - 汇率数据：从国家外汇管理局、新浪财经获取
  - LOF 基金数据：从 API 获取净值和价格
  - 指数 期货 美股数据

- **woody_web_crawler.py**：网页爬虫，作为 API 数据的备用来源

## 数据库模块 (arbcore/database)

数据层负责数据的存储和管理，使用 SQLite 数据库替代了之前的 CSV 文件存储。

- **db_manager.py**：数据库管理类，提供数据库连接、表结构创建和数据操作
- **import_basic_csv.py**：从 CSV 文件导入基础数据到数据库
- **数据库表结构**：
  - `exchange_rates`：汇率数据
  - `futures_settlement`：期货结算价数据
  - `lof_data`：LOF 基金数据
  - `system_health`：系统健康状态

### 数据流向

1. 数据获取模块从外部数据源获取数据
2. 数据处理后存储到数据库
3. 应用层从数据库读取数据进行业务处理
4. 处理结果再存储回数据库或用于展示

应用层包含具体的业务逻辑和用户界面。

### LOFarb (折价套利)

#### LOF00 - 配置中心
- **职责**：Web界面管理基金配置
- **端口**：5001
- **依赖**：无
- **输出**：`lof_config.yaml`

#### LOF011 - 基准日数据
- **职责**：获取汇率、LOF净值和价格、ETF历史价格
- **数据源**：外汇局、Woody、新浪、东财
- **输出**：基础数据到数据库
- **运行时机**：每日盘前

#### LOF012 - 计算静态官方净值LOF历史数据
- **核心算法**：静态官方估值计算

#### LOF013 - 爬woody网页数据
- **职责**：备用的爬虫程序
- **数据源**：Woody网页
- **用途**：静态官方估值计算

#### LOF02 - 实时数据服务
- **职责**：整合所有实时数据源，提供API服务
- **端口**：5000
- **数据源**：QMT、通达信、新浪、东财SSE、IB
- **输出**：SQLite数据库、SSE流、REST API

#### LOF03 - 监控页面生成
- **职责**：生成静态HTML监控页面
- **输入**：数据库中的数据
- **输出**：`lof_monitor.html`

#### LOF01 - 管理面板
- **职责**：Web界面管理系统
- **端口**：5002
- **功能**：启动/停止服务，查看系统状态和日志

在一个完整的 LOF 套利系统生命周期中，有两个关键的黑窗口在后台各司其职：

LOF Backend (5000端口)：这是 LOF02 程序，负责提供实时行情和前端网页渲染（你在看的主面板就是它提供的）。
LOF Admin (5002端口)：这是 LOF01 程序，它是一个独立的指挥中心，负责唤起底层的爬虫去抓取数据或重新计算估值。

### ETFrotate (轮动套利)

### 监控界面与管理面板

- **监控看板**：http://localhost:5000/，实时显示套利机会
- **管理后台**：http://localhost:5002/，用于管理系统和执行任务
- **配置界面**：http://localhost:5001/，用于配置基金信息

### 核心系统架构

#### 1. 后端路由与服务中心 (LOF02_fetch_trade_data.py)
这是整个系统的**核心生命线**。
- **通信层**：采用 Flask 提供 RESTful API 响应单次请求，结合 `flask_socketio` 提供 WebSocket 长连接，实现行情数据的“推/拉”双工模式。
- **守护线程群**：
  - `IBReader`：维护盈透 TWS/Gateway 长连接，负责盘前/夜盘美股 ETF 行情（具备无数据 60 秒兜底降级机制）。
  - `SinaFuturesReader` & `SSEFuturesReader`：维护新浪美股期货和东财SSE国内沪银的实时轮询与 Server-Sent Events (SSE) 流接收。
  - `LOFPriceReader`：负责 A 股 LOF 盘口数据，优先级为：QMT Socket 推送 > 通达信内存快照 > 新浪 API 兜底。

#### 2. 交易网关设计模式 (readers/trade_manager.py)
系统采用了经典的**策略模式与门面模式(Facade)**重构交易模块：
1. **统一入口**：主程序只调用 `trade_manager.send_order(broker, action, symbol, volume, price)`，不关心底层逻辑。
2. **依赖隔离**：所有的 `xtquant`、`socket` 和 `tqcenter` 导入都在该管理器内部进行。即使某个客户端未开启（例如国金 QMT 报错），也不会导致主进程崩溃，而是安全降级。

#### 3. 前端渲染分层架构 (LOF03_generate_monitor_html.py)
为了处理极度复杂的单页面应用 (SPA)，03 主文件演变成了一个“包工头”，具体工作委派给以下三大小弟：

##### 3.1 ConfigManager
- **职责**：解析 `lof_config.yaml`。
- **能力**：处理跨区接力估值配置、基础仓位、汇率及期货动态挂钩锚点。

##### 3.2 DataProcessor
- **职责**：数据清洗与 Pandas 数据框处理。
- **能力**：读取 T-1 甚至更早基准日的官方披露净值、处理节假日跳空缺口、对齐历史 ETF 与期货的基准收盘价，为前端注入高精度基准数据池。

##### 3.3 HtmlGenerator
- **职责**：UI 拼装与样式注入。
- **能力**：管理原生 CSS 与高度复杂的 JS 实时推演逻辑（Sandbox）。实现了完全不依赖后端服务器的“前端 O(1) 复杂度极速沙盘估算”。

#### 4. 健壮性与防崩溃机制
- **静默防刷屏**：底层高频行情推送与终端日志打印解耦，日志实行“冷却时间限制（如30s）”，还开发者一个整洁的控制台。
- **端口占用提示**：Flask 5000 端口如果被历史残留进程占用，系统会通过捕获 `OSError` 拦截报错，并提供中文排障指南。
- **自动降级(Fallback)**：如果 IB 失去夜盘订阅权限，自动平滑退级到 1 分钟历史 K 线快照抓取；如果 QMT 通信失败，自动降级至通达信或只读模式。

## 数据流设计

### 数据获取流程

1. **数据获取流程**：
   - LOF011_daily_updater.py 获取基础数据和基金数据
   - 数据存储到数据库
   - LOF012_calculate_valuation.py 读取数据并计算估值
   - 估值结果存储回数据库

2. **实时数据流程**：
   - LOF02_fetch_trade_data.py 启动实时数据服务
   - 从数据库读取基础数据
   - 实时获取市场数据并计算实时估值
   - 通过 API 和 WebSocket 提供给前端

3. **监控页面生成流程**：
   - LOF03_generate_monitor_html.py 读取数据库中的数据
   - 生成静态 HTML 监控页面
   - 前端通过 API 获取实时数据更新

### 技术特点

1. **模块化设计**：核心功能与业务逻辑分离，便于维护和扩展
2. **数据统一管理**：使用数据库替代 CSV 文件，提高数据管理效率
3. **多源数据获取**：从多个数据源获取数据，提高数据可靠性
4. **实时数据服务**：提供 REST API、WebSocket 和 SSE 接口
5. **可视化监控**：通过网页界面实时监控套利机会
6. **健壮性设计**：多数据源自动切换，单点故障不影响整体
7. **高性能**：异步处理和并行计算，提高系统响应速度
8. **可扩展性**：模块化设计，易于添加新基金或数据源

## 扩展性考虑

### 添加新基金

1. **配置文件更新**：
   - 在 `lof_config.yaml` 中添加新基金的基本信息
   - 配置基金的估值方法、对冲比例等参数

2. **数据获取**：
   - 确保 `LOF011_daily_updater.py` 能够获取新基金的基础数据
   - 确保 `LOF012_calculate_valuation.py` 能够计算新基金的估值

3. **实时监控**：
   - 确保 `LOF02_fetch_trade_data.py` 能够获取新基金的实时数据
   - 确保 `LOF03_generate_monitor_html.py` 能够在监控页面中显示新基金

### 添加新数据源

1. **数据源实现**：
   - 在 `arbcore/fetchers` 目录中实现新的数据源获取类
   - 实现数据获取、解析和处理逻辑

2. **数据源集成**：
   - 在 `data_fetcher.py` 中集成新的数据源
   - 配置数据源的优先级和降级策略

3. **数据存储**：
   - 确保新数据源的数据能够正确存储到数据库
   - 更新数据库表结构（如果需要）


### 接口扩展

1. **API 接口**：
   - 在 `LOF02_fetch_trade_data.py` 中添加新的 API 接口
   - 实现接口的请求处理和响应逻辑

2. **前端功能**：
   - 在监控页面中添加新的功能和数据展示
   - 确保前端能够与新的 API 接口正确交互

## 可靠性设计

1. **数据备份**：定期备份数据库
2. **错误处理**：完善的错误处理机制
3. **监控告警**：系统健康状态监控和告警
4. **数据校验**：对获取的数据进行校验，确保数据质量

通过这种分层架构设计，系统实现了高可用性、可扩展性和可维护性，提供了稳定可靠的基金套利监控服务。
